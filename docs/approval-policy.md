# Approval policy

The agent council and any autonomous Claude Code session consult this
file to decide whether the user must explicitly approve before
proceeding.

Default posture: **err on the side of asking.** A surprised user is more
expensive than a slow agent.

## Approval REQUIRED

Always ask before:

- **Strategy logic changes** — anything under `src/lib/strategies/`,
  including new strategies, threshold tweaks, or removing a strategy
  from the default registry.
- **Scoring logic changes** — `src/lib/engine/scoreEngine.ts`,
  `src/lib/config/constants.ts` (`SCORE_WEIGHTS`, `ACTION_THRESHOLDS`,
  `RISK_PENALTIES`, gate thresholds), `src/lib/engine/riskFilter.ts`.
- **Data adapter changes** — `src/lib/data/adapters/*.ts` (any provider).
- **Deletion of files** — `rm`, `git rm` (with or without `--cached`)
  beyond what `T-012` already executed.
- **Destructive git operations** — `git reset --hard`, `git push --force`,
  `git filter-repo`, `git branch -D`, `git checkout --`, history
  rewrites of any kind.
- **Commits and pushes** — even when the diff is small. The human ships.
- **Dependency changes** — `package.json` `dependencies` or
  `devDependencies` additions / removals / version bumps.
- **Database / schema changes** — not applicable today (Pangzi has no
  DB), but reserved if one is ever added.
- **Live trading, brokerage, or external account access** — explicitly
  out of scope per `SPEC.md` Non-Goals, but doubly forbidden here.
- **CI / GitHub Actions changes** — workflow files under
  `.github/workflows/` and any pre-commit hook installer.
- **Modifying ops files in ways that change the contract** — adding /
  removing required ops files, rewriting `CLAUDE.md` rules,
  re-classifying multiple `DECISIONS.md` entries.

## NO approval required

Safe to act autonomously:

- Reading any file.
- Generating reports under `reports/` (already gitignored).
- Updating docs under `docs/` (other than `approval-policy.md` itself).
- Updating session log entries in `HANDOFF.md` (append-only).
- Adding tests that do not change production code.
- Running tests, typecheck, build, lint, the existing `check:*` scripts.
- Updating `NEXT_ACTION.md` (it is regenerated by the council).
- Updating `reports/agent-council/*.md`.
- Updating `RESEARCH_LOG.md` with new findings (append-only).
- Updating `BUGS.md` to *log* a new finding (changing severity / status
  of an existing entry is borderline — prefer to add a comment and
  flag for human review).

## Borderline (ask if unsure)

- Editing existing test files — fine if the test was wrong; surface a
  comment if the production behavior changed.
- Adding a new npm script — usually fine, but a script that wraps a
  destructive operation is approval-required even before first run.
- Adding a new `data/fixtures/` file — fine if it is small (< 50 KB) and
  deterministic; otherwise ask.
- Touching `.gitignore` — fine to add patterns; ask before removing a
  pattern, since that can re-expose ignored content.

## How an agent uses this file

When the council script picks a task:

1. It inspects `Files:` and the body for keywords matching the "Approval
   REQUIRED" categories above.
2. If any match, `NEXT_ACTION.md` is written with
   `approval_required: true` and a quoted reason.
3. The next Claude Code session reads `NEXT_ACTION.md` first and
   surfaces the chosen task to the user before doing anything beyond
   reading files.

When in doubt, follow the spirit of `CLAUDE.md` — "When in doubt,
default to the most conservative option, and ask before making
non-trivial product changes."
